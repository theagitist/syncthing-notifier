# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A small macOS daemon + helper app that fires notifications when remote
Syncthing devices finish pulling changes from *this* Mac for specific
watched folders. The user wanted Focus-filterable notifications, which is
why we ended up with a real `.app` bundle instead of `osascript display
notification`.

## Layout

| Path | Purpose |
|---|---|
| `notifier.py` | Long-poll daemon (event subscription) — source of truth |
| `notifier_app.swift` | Swift source for the notification helper app |
| `make_icon.swift` | Generates `AppIcon.icns` at install time (pure Core Graphics) |
| `Info.plist.template` | Bundle metadata template (`__BUNDLE_ID__` substituted at install) |
| `launchd.plist.template` | launchd agent template (`__HOME__` and `__BUNDLE_ID__` substituted at install) |
| `config.env.example` | Tracked example; `config.env` (gitignored) holds the real `BUNDLE_ID` |
| `folders.json.example` | Tracked example; `folders.json` (gitignored) holds the real watched folder IDs |
| `install.sh` | Build + deploy to runtime locations |
| `uninstall.sh` | Tear everything down |

## Runtime locations (where install.sh deploys to)

- `~/Applications/Syncthing Notifier.app` — built from `notifier_app.swift`
  + `Info.plist.template` (rendered with `BUNDLE_ID`), with `AppIcon.icns`
  generated by `make_icon.swift`, ad-hoc codesigned, registered via
  `lsregister`
- `~/Library/Application Support/syncthing-notifier/notifier.py` — copy of
  the Python daemon
- `~/Library/Application Support/syncthing-notifier/folders.json` — copy of
  the user's `folders.json`; the daemon reads it via a path relative to
  `__file__`
- `~/Library/LaunchAgents/$BUNDLE_ID.plist` — launchd agent that runs the
  daemon at login (filename derived from `BUNDLE_ID` in `config.env`)
- `~/Library/Logs/syncthing-notifier.log` — combined stdout/stderr

The bundle id comes from `config.env` (`BUNDLE_ID=...`); it is what System
Settings → Notifications and Focus filter on. `config.env` is gitignored,
so the source repo carries no real bundle id — only the placeholder in
`config.env.example`.

## Editing workflow

1. Edit files in this project dir.
2. Run `./install.sh`. It rebuilds the `.app`, copies `notifier.py`, and
   reloads the launchd agent.
3. Tail `~/Library/Logs/syncthing-notifier.log` to verify.

There is no symlink-back arrangement — the runtime files are copies. Always
edit here, never edit the runtime copies, or your changes get overwritten on
next install.

## Why these design choices

- **Swift app, not `osascript`** — `osascript display notification` posts
  under Script Editor's identity, which means Focus can't target *just*
  these notifications. Modern macOS (Sonoma+) is also stricter about legacy
  AppleScript notification flows; an early `osacompile` applet attempt
  silently failed to register with Notification Center.
- **`UNUserNotificationCenter`, not `NSUserNotification`** — the older API
  is deprecated and unreliable on current macOS.
- **`.app` bundle with `LSUIElement = true`** — required so the Swift binary
  can call `UNUserNotificationCenter` (it needs a real bundle id) without
  showing a Dock icon when invoked.
- **Ad-hoc codesigned (`codesign --sign -`)** — no developer ID required;
  good enough for personal local use. `lsregister -f` makes macOS pick up
  the bundle id without rebooting.
- **API key read from Syncthing's config.xml at startup** — keeps the
  daemon in sync if the key rotates, and avoids embedding a secret in the
  project.
- **Filter on `device != myID`** — `FolderCompletion` events fire for both
  inbound and outbound completion; `myID` (from `/rest/system/status`)
  identifies this device's own progress, which we ignore.
- **Prime state at startup with `/rest/db/completion`** — without this,
  a sync that completes between launchd start and the first event would
  generate a spurious notification (we'd see `prev=None → 100`).

## Common modifications

- **Add/remove watched folders** → edit `folders.json` (gitignored), then
  `./install.sh`. Get folder IDs from the Syncthing GUI.
- **Change notification wording** → edit `NOTIFICATION_TITLE_FMT` /
  `NOTIFICATION_BODY_FMT` near the top of `notifier.py`. Available
  placeholders: `{folder_name}`, `{device_name}`, `{completion}`.
- **Different trigger condition** (e.g. notify on partial progress, or only
  for specific devices) → see the `if prev < 100 and fully_synced:` block
  in `main()`.
- **Different Syncthing host/port** → `GUI` constant at the top of
  `notifier.py`.
- **Test the notification path without waiting for a real sync** →
  `python3 ~/Library/Application\ Support/syncthing-notifier/notifier.py --test`.

## Debugging

```sh
# fire a notification directly, bypassing the daemon
"$HOME/Applications/Syncthing Notifier.app/Contents/MacOS/notifier" \
  "Syncthing Notifier" "test"

# restart the daemon without re-running install
launchctl kickstart -k "gui/$(id -u)/$(. ./config.env && echo "$BUNDLE_ID")"

# confirm the agent is loaded
launchctl list | grep syncthing-notifier

# reset notification permission (e.g. after CFBundleIdentifier change)
tccutil reset All "$(. ./config.env && echo "$BUNDLE_ID")"
```

## Gotchas

- After install, `install.sh` fires a test notification through the `.app`
  to trigger the macOS permission prompt. If you missed it, re-run
  `notifier.py --test` or double-click the `.app` once.
- If the daemon dies in a tight loop, `ThrottleInterval=10` in the launchd
  plist forces a 10-second floor between restarts.
- If you change `BUNDLE_ID` in `config.env`, you'll lose the existing
  notification permission entry and have to re-grant (see `tccutil`
  command above).
